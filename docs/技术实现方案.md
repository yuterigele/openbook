# 技术实现方案 · v4.17

> **给面试官 / 技术评审看**：30 分钟理解整个系统怎么搭的，每个关键设计决策为什么这么选。
>
> 对应代码：`github.com/yuterigele/openbook`（v4.17）

## 0. 30 秒全景

```
┌────────────────────────────────────────────────────────────────┐
│  顾客微信 ─→ 企业微信 KF ─→ HTTP/SSE ─→ Pre-check (pool)      │
│                                              │                 │
│                                              ▼                 │
│                          eino ADK Agent (DeepSeek primary,      │
│                              OpenAI/Ark fallback)               │
│                                              │                 │
│   ┌──────┬──────┬──────┬──────┬──────┬──────┐  │                 │
│   │query │create│cancel│list  │handoff│ RAG │  ▼                 │
│   │_sched│_appt │_appt │...   │_human │     │ MySQL 持久化       │
│   └──────┴──────┴──────┴──────┴──────┴──────┘  + Redis 分布式锁  │
└────────────────────────────────────────────────────────────────┘
```

**5 层关键决策**：
1. **CloudWeGo Eino** 而非 LangChain — Go 原生、字节内部验证过、ADK 模式清爽
2. **Function Calling over LangChain Agent** — 业务可控 + 工具可单测
3. **Pre-check 用 worker pool** — bounded concurrency + backpressure
4. **MySQL + Redis 双存储** — 持久化 + 分布式锁
5. **5 层防护**（敏感词 / 意图 / 降级 / RAG / 转人工）— 一个挂了不影响整体

## 1. LLM 接入与降级链

### 1.1 为什么不用 LangChain？

| 维度 | LangChain (Python) | Eino (Go) |
|------|--------------------|-----------|
| 语言 | Python | **Go**（与项目同栈） |
| 性能 | GIL 限制 | 真正的并行 |
| 类型 | 动态 | 强类型（generic `M adk.MessageType`） |
| 部署 | 解释执行 | 单一二进制 |
| 字节内部 | ✗ | ✓（Eino 就是字节出品） |

### 1.2 LLM 降级链实现

`chatmodel/fallback.go`：

```go
func NewModelWithFallback[M adk.MessageType](ctx context.Context) (
    m einomodel.BaseModel[M], used Provider, chain []FallbackEntry, err error,
) {
    providers := DefaultFallbackChain()  // [DeepSeek, OpenAI, Ark]
    for i, p := range providers {
        mm, buildErr := buildProvider[M](ctx, p)
        if buildErr == nil {
            return mm, p, append(chain, FallbackEntry{Provider: p, ...}), nil
        }
        log.Printf("provider %s init failed: %v — trying next", p, buildErr)
    }
    return nil, "", chain, fmt.Errorf("all LLM providers failed: %s", formatChain(chain))
}
```

**3 层保护**：
1. **Init-time fallback**：DeepSeek 启动失败 → 切 OpenAI；服务能起来
2. **Runtime retry**（`helpers/retry.go`）：429 / 5xx / 网络瞬时 → eino 内置重试
3. **Tool-level graceful degradation**：工具失败 → 友好话术（不卡死 Agent）

### 1.3 DeepSeek vs OpenAI vs Ark 的设计权衡

| Provider | 优势 | 劣势 | 何时用 |
|----------|------|------|--------|
| DeepSeek | 中文最便宜、响应快、Function Calling 稳 | 高峰偶发抖动 | 默认主力 |
| OpenAI | 全球最稳、Reasoning 强 | 中文一般、价格高 | DeepSeek 抖动时 |
| Ark | 字节系、合规稳 | 生态略小 | OpenAI 也不稳时 |

## 2. Agent 设计（核心）

### 2.1 Prompt 工程：约束即测试

`internal/agent/agent.go` 的 system prompt 长达 5KB，**所有关键约束都有 v4.x 编号 + 真实事故案例**：

```markdown
【师傅状态信息必须来自工具（v4.16.3 核心，绝不允许幻觉）】
  - 任何关于师傅请假 / 排班 / 具体时段空闲的具体陈述都必须来自工具返回值
  - 真实事故（v4.16.3）：店根本没给老王 7-3 请假，Agent 却对顾客说
    「老王 7-3 上午 10:15-11:15 请假一会儿」——LLM 凭印象凑的时间
```

**为什么这样做**：LLM 行为无法 100% 测，但 prompt 字符串可以测。`agent_prompt_test.go` 验证每次 prompt 修改后关键约束（"创建预约必须告诉预约号"等）还在。

### 2.2 Function Calling 工具设计

**10 个工具，按使用频率排序**：

| 工具 | 调用时机 | 关键约束 |
|------|---------|---------|
| `query_schedule` | **每次创建预约前** | 必调！禁止凭印象 |
| `create_appointment` | 创建预约 | 内部用 Redis 锁防并发 |
| `cancel_appointment` | 取消 | 优先从 history 找 ID |
| `get_appointment` | 改时间前（v4.13.6） | 防 leave 改派后用旧 barber |
| `list_shop_holidays` | 节假日拒绝时（v4.16.2） | 强制拿完整清单防凭印象推 |
| `barber_leave` | 顾客问"为什么没空" | 真实数据，不允许幻觉 |
| `list_barbers` / `list_services` | 顾客问"有哪些师傅" | 简单查询 |
| `mark_noshow` / `mark_completed` | 商户后台 | 状态流转 |
| `handoff_to_human` | 3 类场景才调（v4.13.5） | 写埋点 + dedup 防 LLM 循环调 |

**工具设计原则**：
- 每个工具返回**给 LLM 看的 string**（不是 JSON）
- 错误走 `tools/util.go` 的 `FriendlyError()` 转译成中文
- LLM 拿到 string 错误会自己理解（"时段被占" → "刚被别的顾客抢了"）

### 2.3 工具约束 vs LLM 自由度的边界

LLM 擅长：自然语言理解、对话策略、用户意图
LLM 不擅长：精确时间计算、并发安全、状态一致性

**所以**：
- 业务规则（"节假日拒绝时调 list_shop_holidays"）→ 写在 prompt 里
- 数据一致性（"不能重复创建"）→ 用 Redis 锁在 `create_appointment` 工具里强制
- 状态机（"leave 改派后 barber_name 变化"）→ 在 `get_appointment` 里强制刷新

**核心原则**：能让代码保证的不让 LLM 保证

## 3. 输入防护（5 层）

### 3.1 第 1 层：敏感词审核

`pkg/sensitive/` — 51,345 词，6 大类：

```go
type Result struct {
    Blocked  bool       // 是否拦截
    Category Category   // 命中分类
    Word     string     // 命中的词
    Reason   string     // 给顾客看的拒绝话术
}
```

**性能**：单次 < 1ms（substring contains）

**为什么不分桶**：算法是 `O(词典 × 消息)`，分桶只是改数据布局不影响性能。Benchmark 数据：1 桶 vs 6 桶 性能差距 < 2%（在噪声范围内）。

### 3.2 第 2 层：双层意图分类

`pkg/intent/` — 关键词 + LLM 兜底：

```go
// Layer 1: 关键词白名单（position-based tie-break）
//   "取消刚才那个预约" → 取消（"取消" 位置 0，赢过 "预约" 位置 6）
// Layer 2: LLM 分类兜底
//   "明天下午想去剪个头发" → book（关键词漏了，LLM 兜底）
```

**设计取舍**：
- 关键词层永远在线（免费、零延迟）
- LLM 层只在关键词没命中时触发（< 5% 的输入）
- 节省成本：5000 RPS 每天约 20% 走 LLM 兜底 = 1000 次 × 0.003 元 = 3 元/天

### 3.3 第 3 层：LLM 降级链

见 §1.2

### 3.4 第 4 层：转人工兜底

```go
// 3 类场景才调 handoff_to_human：
//  1) 顾客明确要求找人工
//  2) 顾客需求超出 Agent 能力（投诉/退款/改价）
//  3) 连续 2 轮没识别出意图
//
// 严禁：因顾客语气不好、自己答不上来、怕麻烦就调
// 只调一次（v4.13.5）：per-refID dedup，5 分钟窗口内不重写埋点
```

### 3.5 第 5 层：RAG 知识库

`rag/rag.go` — eino compose.Workflow 4 节点：

```
START{FilePath, Question}
  ↓
[load] os.ReadFile → []*schema.Document
  ↓
[chunk] paragraph splitter → []*schema.Document
  ↓ (并行 MaxConcurrency=5)
[score] ChatModel call × N (并发 5) → []scoredChunk
  ↓
[filter] top-k (e.g., top 3)
  ↓
[answer] synthesize → string
  ↓
END
```

**为什么是 4 节点不是 2 节点**：
- load / chunk 分离：支持长文档分块策略
- score / filter 分离：top-k 与打分策略可独立调整
- 并行 score：20 chunk 用 1s 串行 vs 0.2s 并行（MaxConcurrency=5）

## 4. 并发与状态一致性

### 4.1 并发原语选择

| 场景 | 选用 | 原因 |
|------|------|------|
| Agent 多轮 LLM 调用 | eino ADK | 框架原生 |
| RAG 并行 score | eino BatchNode + MaxConcurrency=5 | 框架内 |
| Pre-check 并行 | **手写 worker pool** | bounded + backpressure + panic recovery |
| 创建预约防并发 | **Redis 分布式锁** | 跨进程 |
| 同 session 多条消息 | 串行（v4.13.2 修 bug） | 之前 4 个 goroutine 互踩 session |

### 4.2 为什么手写 worker pool（不直接 go func）

`pkg/pool/pool.go`：

```go
type Pool struct {
    size    int
    queue   chan Job
    wg      sync.WaitGroup
    closeMu sync.Mutex
    closed  bool
}
```

**核心保证**：
1. **Bounded concurrency**：N 个 worker 不会因为突发流量 OOM
2. **Backpressure**：`Submit` 阻塞 vs `TrySubmit` 立即失败，二选一
3. **Panic recovery**：`defer/recover` + 可注入 `PanicHook`（默认 log + 计数）
4. **Graceful shutdown**：`Close()` 等待 in-flight，停止接新任务

**Benchmark**（敏感词 51K 词 + 50 字消息）：

| 场景 | ns/op |
|------|-------|
| 单桶 51K | 1,403,359 |
| 6 桶 51K | 1,431,326 |
| 短消息（2 字） | 4,219,921 |
| 长消息（500 字） | 19,112,475 |

> 1.4ms/次 × 1000 RPS = 1.4 个核。生产不需要拆桶。

### 4.3 Redis 分布式锁

```go
// tools/create_appointment.go
lockCtx, cancel := context.WithTimeout(ctx, 5*1e9) // 5s
defer cancel()
lock, err := lock.AcquireAppointmentLock(lockCtx, barberID, time)
if err != nil { return FriendlyError(...) }
defer lock.Release()
```

**设计要点**：
- 5s 超时（防 Redis 挂了把 Agent 卡死）
- 锁 key = `barber_id + time`（同一师傅同一时段只允许一个并发）
- defer Release 保证释放

## 5. 流式响应（SSE）

### 5.1 为什么用 SSE 而不是 WebSocket

- 顾客端是微信（移动端 + 弱网）→ SSE 单向推送够用
- 微信小程序/H5 用 SSE 比 WebSocket 省 50% 流量
- hertz-contrib/sse 库现成

### 5.2 心跳保活

```go
stream := sse.NewStream(c)
ticker := time.NewTicker(15 * time.Second)
go func() {
    for range ticker.C {
        _ = stream.Publish(&sse.Event{Data: []byte{}})  // 空心跳
    }
}()
```

**为什么需要**：微信 < 60s 不发数据会断连，每 15s 一发空心跳保活。

## 6. 数据持久化

### 6.1 MySQL 设计要点

- 13 张表（appointments / barbers / services / customers / etc.）
- MsgId 唯一索引做幂等去重（PRD §11.1 P0）
- 软删除（`archived`/`depleted` 状态而非真删）

### 6.2 Redis 用途

- 分布式锁（防并发预约）
- 临时缓存（chat session 数据，**不**持久化）
- 离线场景降级：Redis 挂了 → 锁服务返回无锁句柄，**业务不停**

### 6.3 离线优先存储

`localbk.NewBackend(ctx, &localbk.Config{})` — CloudWeGo Eino 自带，把 checkpoint / state 存到本地 FS。

**生产价值**：顾客突然断网 / 服务重启 / K8s 滚动升级，Agent 状态不丢。

## 9. 关键技术决策对照表

| 决策 | 替代方案 | 为什么选这个 |
|------|----------|---------------|
| Eino (Go) | LangChain (Python) | 同栈、性能、字节内部 |
| Function Calling | LangChain Agent | 工具可单测、约束更可控 |
| 关键词 + LLM 意图 | 纯 LLM 分类 | 95% 输入零成本 |
| 手写 worker pool | goroutine + errgroup | bounded + backpressure |
| Redis 分布式锁 | DB 唯一索引 | DB 唯一索引是兜底；Redis 是热路径 |
| SSE | WebSocket | 移动端友好、省流量 |
| 离线优先 backend | 内存 | 滚动升级 / 断网 / 重启不丢状态 |
| MySQL | PostgreSQL / SQLite | 美业场景读多写少、MySQL 运维更熟 |
| DeepSeek 主力 | OpenAI 主力 | 中文便宜 3-5x |
