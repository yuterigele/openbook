## 0. 一句话价值主张

> 在微信这个 13 亿人都在用的入口里，**1 个 AI Agent 替代 1 个 3500 元/月的前台**，让中小美业门店用 49 元/月就能享受原本要花 5 万/年的 AI 能力。

## 1. 行业现状：3 极分化

### 头部：美团 / 阿里本地生活的"重武器"

- 月活 6 亿+，技术 + 流量 + 地推
- 适合连锁餐饮、SPA、大型医美
- **不适合 1-5 人小门店**（抽佣 15-25%，月单 100 单以下不划算）

### 腰部：美业 SaaS（美约 / 美业帮 / 博卡）

- 4-5 千 / 年，店主学习成本高（"用 SaaS 比雇前台还累"）
- 模板化预约页面、扫码选时间
- **缺点**：顾客需要学新东西（脱离微信），店主功能太多用不上

### 尾部：什么都不用

- 80% 的小门店：**Excel + 微信群 + 老板脑记**
- 痛点最重，付费意愿低（觉得"够用了"）
- **机会**：用 AI 把"什么都不用"升级到"用了反而更省事"

**我们的切入点**：尾部 → 腰部，用 AI 把"老板亲自回微信"变成"AI 自动回"。

## 2. 8 大真实痛点（来自 8 家小门店老板访谈）

### 痛点 1：手工登记 / 记账

```
症状：忙时漏记、忘发卡、统一活动难落实
成本：每月错 1-2 单 × 客单价 80-300 = 月损失 200-600 元
```

### 痛点 2：顾客爽约

```
症状：不预付、不绑卡、不算爽约
成本：旺季单店月损失 3-5 个预约 × 80-300 = 240-1500 元
```

### 痛点 3：无实时库存 / 排班

```
症状：顾客无法在线选空档、换店后丢失历史
成本：顾客"问一下就走"率上升 20-30%
```

### 痛点 4：跨店不互通

```
症状：预约数据散在 5 个私人微信里
成本：月底对账要 1-2 个晚上
```

### 痛点 5：撞单 / 漏单

```
症状：2 个顾客都约 Tony 2 点
成本：顾客体验崩，回头客 -50%
```

### 痛点 6：等候时长无标准

```
症状：顾客不知道前面还有几位
成本：到店即走的"幽灵顾客" 30%+
```

### 痛点 7：顾客画像无沉淀

```
症状：跨员工、跨月无固定消费者档案
成本：营销活动精准度低，复购率差
```

### 痛点 8：违约难管控

```
症状：缺爽约 / 迟到黑名单
成本：定位老板缺席
```

> **老板一天扮演"客服 + 调度 + 财务"角色，每天 1-2 小时耗在微信上**。**这个时间就是我们的目标市场。**

## 4. 技术痛点：如果你要自己做一个 AI 预约系统

下面是实现这个系统时**最容易踩的坑**（也是我们项目已经解决的）：

### 坑 1：LLM 幻觉（v4.16.3 真实事故）

**症状**：
```
顾客：Tony 老师 7 月 3 号有空吗？
Agent：老王 7-3 上午 10:15-11:15 请假一会儿   ← LLM 凭印象编的
商户后台：查无此记录，顾客到店扑空
```

**解法**：
1. Prompt 强约束："任何关于师傅请假 / 排班的具体陈述都必须来自工具返回值"
2. 工具调用前置：`create_appointment` 之前必须先 `query_schedule` 查实时状态
3. 真实事故写进 prompt："v4.16.3 真实事故：店根本没给老王 7-3 请假..."
4. 单测覆盖：每个版本回归测试"师傅名必须用工具返回值"

### 坑 2：师傅 leave 改派后用旧 barber_name（v4.13.6 真实事故）

**症状**：
```
顾客：老王 7-3 改了
Agent：好的，老王 7-3 改好了   ← 但 leave 自动改派给了 Kevin
       实际记录是 Kevin，但回复是老王
```

**解法**：增加 `get_appointment` 工具（v4.13.6），改时间 / 取消前必调，**用本轮真实数据，不用 history**。

### 坑 3：节假日凭印象推日期（v4.16.1 真实事故）

**症状**：
```
顾客：7 月 1 号能约吗？
Agent：7-1 是节假日，7-2 怎么样？   ← 但店 7-1、7-2 都休息
       顾客同意 7-2 后才发现也是假期
```

**解法**：增加 `list_shop_holidays` 工具（v4.16.2），强制先调拿完整清单，**禁止凭 prompt 里的"前后两天"硬推**。

### 坑 4：LLM 死循环调工具（v4.13.5 真实事故）

**症状**：
```
LLM 调 handoff_to_human 拿回"已转人工"
LLM 又调 handoff_to_human（怕没成功）
LLM 又调 handoff_to_human（重复 3-4 次）
event_logs 表里同一顾客多条埋点
```

**解法**：per-refID dedup，5 分钟窗口内只写 1 次埋点。

### 坑 5：撞单并发

**症状**：2 个顾客同时约 Tony 14:00，两个 create_appointment 都成功，但 Tony 1 个时段只能服务 1 人。

**解法**：Redis 分布式锁，key = `barber_id + time`，5s 超时。

### 坑 6：同 session 多消息互踩（v4.13.2 真实事故）

**症状**：3 个 goroutine 并发处理同一 session 的 3 条消息，3 个都看到同一份 history snapshot，回复混乱。

**解法**：`handleKfCallback` 内部同步串行处理，4 个 goroutine → 1 个。

### 坑 7：Sensitive word 误杀 vs 漏杀

**问题**：
- 漏杀：顾客发"你们有没有色情服务" → LLM 答"没有" → 投诉
- 误杀：顾客发"美黑服务"（合法）→ 拦了 → 顾客不满

**解法**（v4.17）：
- 6 大类精细分类（政治 / 色情 / 暴力 / 广告 / 辱骂 / 违法）
- 51,345 词库（Konsheng/Sensitive-lexicon 上游）
- 命中后**给分类话术**（不说"您说了违禁词"），不引发二次试探
- 误杀靠手工精选词兜底（6 大类各 5-10 个高频必中词）

### 坑 8：LLM 降级 + 一致性

**问题**：DeepSeek 挂了换 OpenAI，但 prompt 风格 / Function Calling 兼容性不同，怎么保证行为一致？

**解法**：
- 3 个 provider 都用 OpenAI 兼容 API（DeepSeek = api.deepseek.com/v1 走 OpenAI SDK）
- prompt 不变（"基于工具返回"原则与 provider 无关）
- 测试覆盖：每个 provider 都跑同一组 case

## 6. 我们的护城河

### 短期（6 个月）

1. **5.1 万词敏感词库** + 6 大类精细分类 → 行业最严
2. **真实事故库**：v4.13.5 / v4.16.1 / v4.16.3 都有具体 case，竞品要追需要 1-2 年
3. **微信生态深度集成**：企业微信客服 + 主动通知 + 转人工客服，**比通用 Agent 框架省 3 个月**

### 中期（1 年）

1. **数据飞轮**：使用越多，AI 对师傅档期 / 顾客偏好越精准 → 商家离不开
2. **行业 know-how**：5-10 个真实门店运营 1 年的对话数据 → 比通用 LLM 多 30% 业务准确度
3. **多店 / 连锁支持**：从 1 店 → 5 店 → 50 店，逐级涨客单价（49 → 99 → 199 → 999）

### 长期（2-3 年）

1. **跨行业扩展**：宠物店 / 汽修 / 家政 / 医美，**同一套 AI Agent 框架**
2. **多语种 + 跨境**：服务新加坡 / 东南亚华人市场
3. **插件市场**：第三方开发者上架美业相关 skill

## 7. 不做什么

| 不做 | 原因 |
|------|------|
| 不做美团 / 阿里那种"重平台" | 流量是 10 年级战，我们没那个本钱 |
| 不做"通用 Agent 框架" | 已有 LangChain / Eino 等成熟方案，差异化在垂直深度 |
| 不做"美业 SaaS 全套"（收银 / 库存 / 营销） | 已有美约 / 美业帮等，**我们只做 AI 预约这点** |
| 不做"低代码 / 零代码" | 老板不是技术人，但 1 个 AI 助手比 100 个模板更省事 |
