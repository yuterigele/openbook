## 2. 企业微信接入层

### 2.1 消息适配层设计

统一回调入口，根据 `MsgType` 和来源信息，将微信客服与客户联系两条通路的消息归一化为内部 `InternalMessage` 结构。

```go
// InternalMessage 内部统一消息格式
type InternalMessage struct {
    Source       string    // "wecom_kf" | "wecom_customer"
    MsgID        string    // 用于幂等去重
    ExternalUserID string  // 顾客统一标识
    ShopID       string    // 关联店铺（由 Agent 根据来源解析）
    Content      string    // 文本内容
    CreateTime   time.Time
}
```

适配器接口：

```go
type MessageAdapter interface {
    Adapt(raw []byte) (*InternalMessage, error)
}
```

实现两个适配器：

- `WecomKfAdapter` — 处理微信客服消息体
- `WecomCustomerAdapter` — 处理客户联系消息体

**关键实现点**：

- 消息签名验证与解密前置，在进入适配器前完成
- 每条消息用 `MsgID` 做幂等校验，已处理过的直接返回 200 不回 Agent

### 2.2 消息去重

在 `storage` 层维护一张 `msg_dedup` 表，`msg_id` 列建唯一索引。每次新消息到来时先 `INSERT IGNORE`，根据影响行数判断是否已处理。

## 3. Agent 会话管理

### 3.1 Redis 会话存储

每个顾客的对话上下文以 `external_userid` 为粒度存储在 Redis 中。

- **Key 格式**：`session:{external_userid}`
- **Value**：JSON 序列化的最近 N 轮对话（LLM 标准 `messages` 数组格式）
- **TTL**：30 分钟，每次新消息到来时刷新

### 3.2 会话生命周期

```
新消息到达 → 从 Redis 读上下文 → 追加新消息 → 调用 LLM
    → 将 LLM 回复追加进上下文 → 写回 Redis（刷新 TTL）
```

30 分钟无消息则会话自动过期，下次顾客发消息时从零开始新会话。

## 4. Agent 工具集设计

### 4.1 工具总览

| 工具名 | 用途 | 关键参数 | 返回值 |
|--------|------|---------|--------|
| `create_appointment` | 创建预约 | date, time, barber_name, service_name | 预约确认信息 |
| `query_schedule` | 查询空闲时段 | date, barber_name (可选) | 可约/请假/已约满三段 |
| `cancel_appointment` | 取消预约 | appointment_id 或 (date+time+barber) | 取消结果 + 是否计罚 |
| `list_barbers` | 列出可预约理发师 | 无 | 理发师列表 + 今日请假标记 |
| `list_services` | 列出可用服务 | 无 | 服务名称 + 预估时长 |
| `barber_leave` | 查询理发师请假 | barber_name, date | 请假区间 + 原因 |
| `handoff_to_human` | 转人工 | customer, reason, last_message | 转接成功提示 |

### 4.2 工具实现规范

所有工具统一放在 `tools/` 目录下，每个工具一个文件，实现 Eino 的 Tool 接口：

```go
type Tool interface {
    Info() *schema.ToolInfo
    InvokableRun(ctx context.Context, argumentsInJSON string,
        opts ...tool.Option) (string, error)
}
```

每个工具文件中包含：

1. 工具描述（`Info()` 返回的 `Desc` 字段，用中文撰写，供 LLM 理解何时调用）
2. 参数解析与校验
3. 核心逻辑（调用 `storage` 层）
4. 返回结果序列化（人类可读的自然语言文本，LLM 直接用于回复顾客）

### 4.3 关键工具流程

**`create_appointment` 执行流程：**

```
参数解析 → 校验服务是否存在 → 校验理发师是否存在
    → 检查理发师是否在所选时段请假（调用 IsBarberOnLeaveAt）
    → Redis 分布式锁（key: lock:appointment:{barber_id}:{start_at}）
    → 检查时段是否已被占用（MySQL 查重）
    → 写入预约记录
    → 释放锁
    → 返回确认信息
```

**`query_schedule` 渲染格式：**

采用三段式渲染，让 LLM 能直接判断应建议“换时间”还是“换师傅”：

```
理发师 Tony 在 2026-06-22 的可预约时段：
  09:00, 09:30, 10:00, 10:30, 11:00, 11:30, 13:30, 16:30, 17:00, 17:30, 18:00
师傅请假占用：14:00-16:00（体检）
（这些时段是师傅临时请假，建议换时间或换其他理发师）
其余 2 个时段均已被预约。
```

### 5.2 数据库选型说明

- **MySQL 8.0** 为主存储，存储所有业务数据
- **Redis 7** 用于会话缓存（30 分钟 TTL）和分布式锁（10 秒 TTL）
- 时间统一存为 UTC，查询时根据 `shops.timezone` 转换为本地时间

## 6. 并发控制

### 6.1 双重保障策略

| 层级 | 机制 | 作用 |
|------|------|------|
| 应用层 | Redis SETNX 分布式锁 | 乐观锁前置，快速拒绝并发冲突 |
| 数据库层 | MySQL 唯一索引 `uk_barber_time` | 最后防线，兜底防止脏数据写入 |

### 6.2 分布式锁实现

```go
func AcquireAppointmentLock(ctx context.Context, barberID string, startAt time.Time) (bool, error) {
    key := fmt.Sprintf("lock:appointment:%s:%d", barberID, startAt.Unix())
    ok, err := redisClient.SetNX(ctx, key, "1", 10*time.Second).Result()
    return ok, err
}
```

- **Key 粒度**：`lock:appointment:{barber_id}:{start_at}`，精确到理发师+时段
- **TTL**：10 秒，覆盖正常写入耗时（远小于 1 秒），TTL 用于防止死锁
- 预约写入成功后主动释放锁，失败时锁在 10 秒后自动过期

## 7. 定时任务

### 7.1 任务清单

| 任务名 | 频率 | 职责 |
|--------|------|------|
| 预约提醒 | 每 1 分钟 | 扫描 `start_at` 在 2 小时后的预约，发送微信提醒 |
| 爽约扫描 | 每 5 分钟 | 扫描 `start_at` 已过 30 分钟仍为 `active` 的预约，标记为 `noshow`，累计顾客爽约次数 |
| 请假过期 | 每 1 分钟 | 将 `end_at < now` 的 `active` 请假自动转为 `expired` |
| 续费生命周期 | 每 1 小时 | 检查店铺订阅状态，按 D+3/D+15/D+25 等节点触发提醒 |
| D+15 报告邮件 | D+15 当天 10:00 | 生成使用报告并通过 SMTP 发送给商户 owner |

### 7.2 定时任务框架

使用 `robfig/cron` 库管理定时任务，在 `main.go` 中统一注册启动。

## 8. 通知层

### 8.1 企业微信消息发送

两类消息接口：

| 消息类型 | 接口 | 使用场景 |
|---------|------|---------|
| 客服消息 | 微信客服 API | 顾客正在对话中，Agent 主动回复 |
| 模板消息 | 企业微信模板消息 | 预约提醒、系统通知等非会话内消息 |

封装为 `WecomNotifier` 接口：

```go
type WecomNotifier interface {
    SendText(ctx context.Context, externalUserID, content string) error
    SendTemplate(ctx context.Context, externalUserID, templateID string, data map[string]string) error
}
```

### 8.2 邮件发送

用于 D+15 使用报告等商户侧通知。通过 SMTP 协议发送 HTML 格式邮件。

- 默认端口 465（SSL）
- 若未配置 SMTP，则自动降级为 NoopSender（仅打日志，不报错）
- 收件人通过环境变量 `REPORT_TO` 配置

## 9. 错误处理与降级

### 9.1 工具调用异常处理

| 异常场景 | 处理策略 |
|---------|---------|
| 工具返回错误（如时段冲突） | 错误信息直接传递给 LLM，让 LLM 生成对顾客的自然语言解释 |
| 数据库暂时不可用 | 返回明确错误信息，LLM 告知顾客“系统暂不可用，请稍后再试” |
| Redis 不可用 | 降级为无会话模式（单次问答），分布式锁改用 MySQL 唯一索引兜底 |
| LLM API 超时/限流 | 重试 1 次，仍失败则返回默认话术 |

### 9.2 转人工兜底

当以下情况发生时，Agent 调用 `handoff_to_human` 工具：

- 顾客明确要求转人工
- 连续 2 轮无法识别顾客意图
- 需求超出工具能力范围（退款、投诉等）

`handoff_to_human` 写一条埋点事件，商户在后台可见待处理列表，主动联系顾客。
