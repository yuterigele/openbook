# AGENTS.md

本文件用于指导在本仓库中工作的 AI 编程代理。除用户的明确要求外，修改代码时应遵守以下约定。

## 项目概览

OpenBook 是一个面向美发门店的微信预约 Agent MVP。用户消息经企业微信或本地会话进入服务端，由 Agent 识别意图并调用受限业务工具，最终通过存储层完成查询、预约、改约、取消和转人工等操作。

主要技术栈：

- Go 1.25、CloudWeGo Eino、CloudWeGo Hertz
- MySQL 8、Redis 7
- DeepSeek、OpenAI、Ark 模型适配及降级链
- 静态管理端与 Vitest 前端单测
- Docker Compose 本地运行

## 目录职责

- `main.go`、`server/`：应用入口、HTTP/消息处理、限流及回复流程
- `internal/agent/`：Agent 编排、提示词、模型与工具循环
- `tools/`：模型可调用的预约业务工具及业务校验
- `storage/`：数据模型、仓储、权限约束及事务逻辑
- `api/`、`auth/`：管理 API、认证与授权
- `chatmodel/`：模型适配、桩实现及降级逻辑
- `wecom/`：企业微信接入、验签、加解密及去重
- `cron/`、`notify/`：定时任务与通知
- `static/`：聊天页和管理端静态页面
- `web/`：前端工具函数与 Vitest 测试
- `cmd/`、`scripts/`：运维、迁移、种子数据及构建工具
- `docs/`：产品、部署、基准与工程复盘文档

## 核心安全边界

- 不要给顾客消息开放文件系统、Shell、任意 SQL 或后台管理能力。
- Agent 只能调用显式注册且完成服务端校验的业务工具；不能依赖提示词代替授权或数据校验。
- 门店 ID、顾客身份、会话身份等可信字段必须由服务端上下文注入。忽略模型或客户端自行提供的同名身份字段。
- 查询、取消、改约等操作必须同时校验门店隔离、资源归属和调用者权限，防止跨店或跨顾客访问。
- 创建或改约要保持事务与并发保护，不能绕过现有 Redis 锁、数据库约束或冲突检测。
- 模型调用失败或结果不可信时应安全降级；不得在无法确认业务结果时写库或回复“操作成功”。
- 不提交 `.env`、API Key、企业微信密钥、数据库口令、真实手机号或其他敏感数据。示例配置只写入 `.env.example`，并使用无效占位值。

## 修改原则

- 先阅读目标包及其相邻测试，遵循现有接口、命名和错误处理方式，避免无关重构。
- 业务规则应落在 `tools/` 或 `storage/` 等确定性代码中，而不是仅写进 Agent 提示词。
- 涉及时间语义时使用项目约定的 `Asia/Shanghai` 运行时上下文；不要把固定示例日期当成当前日期。
- 修改存储行为时同步检查事务、幂等性、软删除/状态过滤、门店隔离和 SQLite 测试兼容性。
- 修改消息入口时同步检查重复投递、限流、敏感内容、客服回调及 mock/真实回复模式。
- 新增模型或工具能力时保持白名单注册，并为成功、失败、越权和边界输入补充测试。
- 修改公开行为、环境变量、部署步骤或重要业务规则时同步更新 `README.md`、`.env.example` 或 `docs/`。
- 保持变更小而聚焦，不覆盖工作区中与当前任务无关的用户修改。

## 格式与测试

Go 代码修改后执行：

```bash
gofmt -w <changed-go-files>
go test ./<changed-package> -count=1
```

跨包或高风险修改应执行完整测试：

```bash
go test ./... -count=1
```

预约归属和取消权限相关修改至少执行：

```bash
go test ./tools -run 'Test(GetAppointment|CancelAppointmentTool|E2E_S2_CancelAppointment)' -count=1
go test ./internal/agent ./server -count=1
```

前端 JavaScript 修改后执行：

```bash
cd web
npm test
```

涉及容器、环境变量或启动流程时，检查 Compose 配置并做相应冒烟验证：

```bash
docker compose config
docker compose up --build
```

如果受环境、凭据或外部服务限制无法运行某项测试，应明确说明未验证的内容和原因，不要把未执行描述为已通过。

## 完成标准

- 实现满足用户需求，且没有扩大权限或破坏多租户/身份边界。
- 相关测试已新增或更新，修改过的代码已格式化。
- 已运行与风险相称的测试，并如实报告结果。
- 文档与配置示例和实际行为一致。
- 未引入秘密信息，也未改动、删除或回滚无关的现有工作区内容。
